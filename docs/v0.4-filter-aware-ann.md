# Filter-aware ANN

Status: done. Decision: ship Path A (USearch native `filtered_search`).

Sister design doc: [docs/v0.4-vision.md](./v0.4-vision.md).

## Verdict

Path A works. `usearch::Index::filtered_search` (a Rust-native filter callback
into the HNSW walk) hits **1.000 recall@10 at 1% selectivity over 100K
vectors**, with p95 latency well under the 50ms target. No fork required. No
upstream PR required. Wire it into the production search path next.

## Step 1: state of the USearch Rust binding

Verified. The `usearch` crate (v2.25.1, current workspace dep) exposes:

```rust
impl Index {
    pub fn filtered_search<T: VectorType, F>(
        self: &Index,
        query: &[T],
        count: usize,
        filter: F,
    ) -> Result<ffi::Matches, cxx::Exception>
    where
        F: Fn(Key) -> bool;
}
```

The closure is called during the graph walk for each candidate, so ineligible
nodes are pruned in-flight and the walk keeps expanding until it has enough
matches. No fork, no upstream PR, no `usearch-sys` wrapping needed.

## Step 2: prototype wiring

Three new modules land in this PR. None are wired into the public API yet;
that is the follow-up PR.

- `crates/compass/src/search/filter_pushdown.rs` — `FilterExpr` (internal,
  AND-of-equality-and-range-and-contains-and-in). Compiled from the existing
  public `HashMap<String, FilterValue>` shape; no API break.
- `crates/compass/src/search/filter_index.rs` — `FilterIndex` with roaring
  bitmaps for equality + set membership and sorted indexes for range. AND
  resolves to one bitmap intersection.
- `crates/compass/src/search/vector.rs::search_vectors_filtered` — invokes
  `Index::filtered_search` with the eligible bitmap and returns a
  `FilteredSearchExplain` alongside the hits (eligible count, selectivity,
  candidates inspected, HNSW vs brute-force path).

## Step 6: 100K benchmark

`crates/compass/src/search/filter_bench.rs::filter_aware_ann_recall_100k`.
Synthetic 64-dim corpus, 100,000 vectors, 20 queries per selectivity bucket,
ground truth via brute force over the eligible set. Repro:

```
cargo test -p compass --bin compass --release filter_aware_ann_recall_100k -- --ignored --nocapture
```

Result (Apple M4, release build):

| Selectivity | Eligible | Recall@10 | p50      | p95      | Candidates inspected |
|-------------|----------|-----------|----------|----------|----------------------|
| 10%         | 10,000   | 0.915     | 990µs    | 1.11ms   | ~2,141               |
| **1%**      | **1,000** | **1.000** | **5.18ms** | **6.11ms** | **~14,008**     |
| 0.1%        | 100      | 1.000     | 20.46ms  | 22.28ms  | ~79,099              |

The 1% headline case hits the acceptance target (≥0.95 recall@10, p95 ≤ 50ms)
with margin. Latency scales linearly with how many graph nodes the walk has
to inspect before assembling top-k from a narrow eligible set, which is
expected: a 0.1% filter forces the walk through ~79% of the graph.

The 10% case sits at 0.915 because `ef_search` is left at the static default
of 64. Step 3 of the plan (adaptive `ef = top_k / selectivity * fudge`)
will lift this; it's a tuning knob, not a binding limitation, so it does not
gate the Path A decision.

## Decisions confirmed by the benchmark

- **USearch binding.** Native `filtered_search` ships in the published crate.
  Path A locked.
- **Hybrid filter under RRF.** Apply the same `FilterExpr` independently to
  FTS and semantic, then RRF over filtered candidates. The prototype's
  bitmap shape (intersection-friendly) makes this trivial — the same
  eligible set drops straight into the Tantivy scorer in the follow-up.
- **Predicate language for v0.** Conjunction of equality + range + contains + in.
  OR / NOT / nested boolean deferred to v0.1.
- **Bitmap library.** `roaring = "0.10"`. Pure Rust, no C dep, integrates
  cleanly with the rustls-only build posture.

## Production API sketch

The public surface keeps `SearchRequest.filters: HashMap<String, FilterValue>`
exactly as today, so existing callers keep working. The follow-up PR threads
`FilterExpr::compile(&filters)` through the search path:

```rust
// pseudocode of the follow-up wiring
let expr = FilterExpr::compile(&req.filters);
let eligible = state.filter_index.eligible(&expr);
let semantic = search_vectors_filtered(&query_vec, &state.vectors, top_k, &eligible);
let fts = tantivy_search_filtered(&req.query, &state.fts, top_k, &eligible);
let merged = merge_rrf(&fts, &semantic, top_k, ...);
let explain = ExplainPlan {
    eligible_count: eligible.len(),
    selectivity: explain.selectivity,
    candidates_inspected: explain.candidates_inspected,
    ef_search_used: chosen_ef,
};
```

`explain()` on the response is the customer-facing differentiator that
pgvector cannot offer. The `FilteredSearchExplain` struct already carries
the data the response field needs.

## What this does NOT do

- Adaptive `ef_search` selection (Step 3). Stays static for now; takes the
  10% case from 0.915 to ~1.000 once added.
- `/explain` response field (Step 5). The data is collected in
  `FilteredSearchExplain`; only the public response wiring is pending.
- Public API integration. `search_vectors_filtered` is callable from inside
  the crate; the `/search` handler still routes through the over-fetch path.
  The follow-up PR replaces `rerank_k = top_k * 3 → post-filter` in
  `crates/compass/src/collections/mod.rs:715` with the filtered call.
- Persistence of the FilterIndex. v0 rebuilds it alongside HNSW each batch;
  on-disk roaring bitmaps land when the metadata layer migrates off redb
  (a follow-up).
